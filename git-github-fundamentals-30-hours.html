01
Introduction to Version Control
What version control is, why every developer uses it, and getting Git & GitHub installed and ready.
HOURS 1–6
HOUR 01

What is Version Control, and Why Do You Need It?

Version control is a system that automatically tracks every change made to a file, so you can see what changed, who changed it, and go back to any earlier version — without ever renaming a file.

Real-world exampleYou've all done this: Assignment_final.docx, Assignment_final2.docx, Assignment_FINAL_use_this_one.docx. Version control replaces that mess — one file, full history, zero confusion.
Why it matters for youAlmost every coding interview today assumes you know Git. It's treated as a basic skill, like typing.
HOUR 02

Local, Centralized & Distributed Version Control Systems

Local VCS keeps history only on your own machine — like a personal diary. Centralized VCS stores all history on one shared server everyone connects to — like one library book the whole class shares. Distributed VCS (what Git is) gives every person a full copy of the entire project history on their own laptop — like everyone having their own complete photocopy of that book.

Real-world exampleIf a centralized server crashes, nobody can work. With Git, even if GitHub goes down for a day, you still have the entire project history sitting safely on your own computer.
HOUR 03

Why Git and GitHub? The Real Benefits

Git is the tool that runs on your computer and tracks history, even fully offline. GitHub is a website that stores your Git projects online so you can back them up, collaborate with others, and showcase your work publicly.

Real-world exampleDuring placement interviews, recruiters frequently ask for your GitHub profile link — it works like an online resume made entirely of real, working code.
HOUR 04

Installing Git on Windows & the config Command

Installing Git is just like installing any other software — download it from git-scm.com and click through the setup. The first thing you do afterward is tell Git who you are, using git config, so every change you make is "signed" with your name.

terminal
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"
Real-world exampleIt's exactly like writing your name on the first page of a college notebook — it records who made every entry that follows.
HOUR 05

Getting Help Through the help Command

Git has a built-in manual for every single command, so you never have to leave the terminal to remember how something works.

terminal
$ git help
$ git help commit
$ git commit --help
Real-world exampleLike having a reference manual sitting open next to you during a practical exam — always available, no internet required.
HOUR 06

Creating a GitHub Account & First Look Around

Sign up at github.com, choose a profile picture, and explore the dashboard — your repositories, notifications, and profile page.

Why it matters for youPick your username carefully — something like firstname-lastname rather than a gamer-tag. This becomes part of your professional identity that recruiters will see.
02
Getting Started with Git
Repositories, the staging area, your first commit, and syncing with GitHub.
HOURS 7–12
HOUR 07

Local vs. Remote Repositories

A "repository" (repo) is simply a project folder that Git is tracking. The local repo lives on your own computer; the remote repo is the copy stored on GitHub's servers online.

Real-world exampleThe local repo is your rough notebook at home; the remote repo is the final, neat copy submitted to the college office that everyone else can access too.
HOUR 08

git init — Starting Your First Repository

Running git init inside any folder turns it into a Git-tracked project by creating a hidden .git folder that quietly stores all the history.

terminal
$ mkdir my-college-project
$ cd my-college-project
$ git init
Real-world exampleLike opening a fresh lab logbook the moment you start an assignment, so every step gets recorded from day one.
HOUR 09

git status & git add — The Staging Area

git status shows which files changed. git add moves those changes into a "staging area" — a waiting room before anything is permanently saved.

terminal
$ git status
$ git add app.py
$ git add .   // stage everything that changed
Real-world exampleThink of online shopping — git add is adding items to your cart. You haven't checked out yet, you're just selecting what to save.
HOUR 10

git commit — Saving a Permanent Snapshot

git commit takes everything in the staging area and saves it as a permanent, labelled checkpoint in your project's history, along with a message describing what changed.

terminal
$ git commit -m "Added login page"
Real-world exampleIt's a save-point in a video game — you can always return to that exact moment later. A clear message like "Fixed divide-by-zero bug in calculator app" helps your future self and teammates instantly.
HOUR 11

git clone & git remote

git clone downloads a full copy of an existing GitHub repository onto your computer. git remote is how your local repo stays connected to that GitHub address.

terminal
$ git clone https://github.com/username/project.git
Real-world exampleWhen a professor shares a starter-code repository link for a lab, "cloning" it gives you your own working copy on your laptop in one command.
HOUR 12

push, pull, fetch & Your First Repository

git push uploads your local commits to GitHub. git pull downloads new commits and merges them in automatically. git fetch downloads new commits but lets you review them first, before merging.

terminal
$ git push origin main
$ git pull origin main
$ git fetch origin
Real-world exampleSubmitting an assignment via GitHub Classroom is literally git push. Checking if a teammate added new code is git pull. Today's hands-on: create your first GitHub repository, connect it, and push your first commit end-to-end.
03
Branching and Merging
Working safely in parallel, combining work back together, and resolving conflicts.
HOURS 13–18
HOUR 13

What is a Branch, and Why Do We Need One?

A branch is an independent, parallel copy of your project where you can experiment freely, without touching the main, working version at all.

Real-world exampleLike writing a rough draft of an essay on a separate page while keeping your final neat copy untouched — if the draft doesn't work out, you simply discard that page. A team building a college management app keeps a stable main branch for demos, while one student tries a risky new attendance feature on its own branch.
HOUR 14

Creating and Switching Branches

git branch creates a new branch, and git switch (or the older git checkout) moves you into it.

terminal
$ git branch feature-login
$ git switch feature-login
// or, shortcut for both steps:
$ git switch -c feature-login
Real-world exampleNaming branches clearly — like feature-login or bugfix-navbar — instantly tells teammates what you're working on, no explanation needed.
HOUR 15

Working on a Feature Branch

Once you're on your new branch, you code, add, and commit exactly as before — but the main branch stays completely safe and unaffected the entire time.

Real-world exampleFour students on one final-year project each take their own branch — search-feature, profile-page, payment-module, admin-panel — and work in parallel without stepping on each other's code.
HOUR 16

git merge — Combining Branches Back Together

Once a feature is tested and working, git merge brings it back into the main branch.

terminal
$ git switch main
$ git merge feature-login
Real-world exampleOnce the new login feature is confirmed working on its own branch, merging it brings it into the version the whole team actually uses.
HOUR 17

Merge Conflicts — and How to Resolve Them

A conflict happens when two branches changed the same line of the same file differently, and Git can't decide which version to keep — so it asks you.

Real-world exampleYou and a teammate both edited line 10 of index.html differently. Git marks both versions with <<<<<<< and >>>>>>> markers; you open the file, decide what the final line should actually be, then commit the resolved version.
HOUR 18

Deleting Branches — Cleaning Up

Once a branch has been merged and is no longer needed, delete it to keep the project tidy.

terminal
$ git branch -d feature-login
Real-world exampleJust like deleting a rough-draft file once your assignment is finalised and submitted — there's no reason to keep the clutter around.
04
Advanced Git Commands
Reading history, comparing changes, and undoing mistakes the right way.
HOURS 19–24
HOUR 19

git log — Viewing Your Project's History

git log lists every commit ever made — author, date, and message — your project's complete diary.

terminal
$ git log
$ git log --oneline --graph
Real-world exampleBefore a viva on your own project, scrolling through git log reminds you exactly what you built and when — a built-in memory aid.
HOUR 20

git diff — Comparing Changes Line by Line

git diff shows exactly which lines were added or removed between two versions — additions and removals highlighted clearly.

terminal
$ git diff
Real-world exampleBefore committing, run git diff to double-check you didn't accidentally leave a stray print("testing") or commented-out code in your assignment.
HOUR 21

git stash — Saving Unfinished Work Temporarily

git stash tucks away your uncommitted, half-finished changes safely, giving you a clean slate — without forcing you to commit unfinished work.

terminal
$ git stash
$ git stash pop   // bring it back later
Real-world exampleLike sweeping unfinished homework into a drawer when a guest suddenly arrives, then taking it back out exactly as you left it. If your professor needs an urgent fix on main while you're mid-feature, stash your work, fix the bug, then resume.
HOUR 22

git reset — Undoing Commits

Three modes: soft (undo the commit, keep changes staged), mixed — the default (undo the commit and unstage, but keep your file edits), and hard (discard everything — use with real care).

terminal
$ git reset --soft HEAD~1
Real-world exampleYou committed a feature by mistake before it was ready. git reset --soft HEAD~1 undoes just the commit, letting you fix it and recommit cleanly.
HOUR 23

Working with Tags — Marking Important Versions

A tag is a permanent label marking a meaningful point in history — usually a release version, like v1.0.

terminal
$ git tag v1.0
$ git push origin v1.0
Real-world exampleWhen you submit your final-year project, tag that exact commit as final-submission, so you and your evaluator can always find that precise version later, even after you keep coding past it.
HOUR 24

revert vs. reset vs. checkout

git revert creates a new commit that undoes an earlier one — safe for shared/team work since history stays intact. git reset rewrites history — only safe on your own local, unshared commits. git checkout (or restore) lets you view old versions or discard changes to specific files, without touching history at all.

Real-world exampleIf a bug was already pushed to a shared GitHub repo your whole team uses, use git revert — it won't erase history your teammates already have. If it's just your own local mistake, not yet pushed, git reset is perfectly fine.
05
Introduction to GitHub
Repositories, issues, pull requests, forking, and the real team workflow used at companies.
HOURS 25–30
HOUR 25

Navigating the GitHub Interface

A guided tour of a repository page: the Code tab, README display, Issues, Pull Requests, commit history, file structure, and the green "Code" button used for cloning.

Why it matters for youBefore an internship interview, recruiters often open your GitHub profile and click straight into your repositories — knowing how to make this look clean and organised genuinely matters.
HOUR 26

Creating & Managing Repositories on GitHub

How to create a new repository directly on GitHub, choose public or private, add a README.md describing the project, and a .gitignore that tells Git which files to ignore (like temporary build files).

Real-world exampleA good README with a project description, screenshots, and setup steps is often the very first thing a recruiter or teammate reads — make it count.
HOUR 27

Issues — Tracking Bugs and Tasks

An "Issue" on GitHub is a ticket for a bug, task, or idea that needs attention — with comments, labels like bug or enhancement, and assignees.

Real-world exampleA team building a library-management system opens issues like "Search bar not working on mobile" and "Add fine-calculation feature," then assigns each one to a team member — a shared to-do board for the entire project.
HOUR 28

Pull Requests — Proposing and Reviewing Changes

A Pull Request (PR) formally asks to merge your branch's changes into another branch — usually main — giving teammates the chance to review and comment before it's accepted.

Why it matters for youThis is exactly how real companies and open-source projects work: nobody pushes straight to main; every change goes through a Pull Request and review first. Practising this in college prepares you directly for industry workflow.
HOUR 29

Forking Repositories

Forking creates your own personal copy of someone else's repository, under your own account, which you can freely modify without affecting their original project.

Real-world exampleWant to contribute to a popular open-source project, or build on a senior's old project as your own mini-project? Fork it, make your changes, and optionally send a Pull Request back to the original.
HOUR 30
