The Soluation: Version Control

Version control is a system that automatically tracks every change made to a file, so you can see what changed, who changed it, and go back to any earlier version — without ever renaming a file.

Local, Centralized and Distributed Version Control Systems

Local VCS: A simple database on your computer that tracks changes. No collaboration features. Example: RCS (1980s). Rarely used today. Keeps history on your local machine only. If your computer crashes, you lose everything. - like a personal diary.

Centralized VCS: A single server stores all versions of the project. Developers check out files from this central place. Example: CVS, Subversion (SVN). Collaboration is possible, but if the central server goes down, no one can save changes or access history. - like one library book the whole class shares.

Distributed VCS: Every developer has a complete copy of the entire project history on their own machine. Example: Git, Mercurial. Collaboration is seamless, and there's no single point of failure. - like everyone having their own complete photocopy of that book.

Distributed VCS Types

1. Git & Github: Created by Linus Torvalds in 2005 for Linux kernel development. Fast, efficient, and widely adopted. Uses a unique snapshot-based model. - like a supercharged version of a personal diary with branching timelines.

2. Mercurial: Similar to Git but with a simpler design. Less popular than Git but still used in some projects. - like a more straightforward diary without branching.

3. Fossil: An integrated VCS that includes bug tracking and wiki features. Used by SQLite. - like a diary that also has built-in to-do lists and notes.

4. Bazaar: Developed by Canonical (Ubuntu). Focuses on simplicity and ease of use. Less popular than Git. - like a diary designed for beginners.

5. Perforce: A commercial VCS used in large enterprises and game development. Handles large binary files well. - like a diary that can also store photos and videos.

6. BitKeeper: A commercial VCS that was used for Linux kernel development before Git. - like a diary that was once popular but is now mostly replaced.

7. Darcs: A VCS based on a unique patch-based model. Less popular than Git. - like a diary that focuses on the individual changes rather than snapshots.

Azure Git - A cloud-based Git repository service provided by Microsoft Azure. It offers Git version control along with additional features for collaboration, continuous integration, and deployment. - like a diary that is stored in the cloud with extra tools for team projects.

What is Git?

Git is a version control system (VCS) — a software tool that lives on your computer and automatically tracks every single change made to your files. It's like having a detailed diary for your project where every entry is timestamped, labeled, and reversible.

In simple terms: Git saves "snapshots" of your project at different points in time. Each snapshot (called a "commit") records who made the change, when they made it, what exactly changed, and why (via a message). You can view the entire history, jump back to any earlier version, or see exactly what changed between two versions.

Git is a program you install on your computer. It tracks changes to your files locally on your machine. Simple Analogy: Git is like your personal notebook where you write your work and keep every draft. You can flip back to any page, see who wrote what, and even erase mistakes without losing the entire notebook.

Think of it as:

  • A time machine: Go back to any previous version instantly
  • A team notebook: See exactly who changed what, when, and why
  • A safety net: Experiment freely without breaking working code
  • An insurance policy: Every change is backed up forever

What is GitHub?

GitHub is a website (github.com) where you store your Git projects online. It's like Google Drive, but specifically designed for code and collaboration.

Git

Software on your computer that tracks changes locally. Works completely offline.

GitHub

Website where you backup your Git projects online. Enables team collaboration and serves as your developer portfolio.

✓ Key Insight

You can use Git without GitHub (just tracking files locally). But you can't use GitHub without Git (GitHub uses Git to work). They're complementary tools.

What Files Does Git Create?

When you run git init, Git creates a hidden folder called .git inside your project. This folder contains: 

your-project/
├── src/
│   └── main.py
├── README.md
└── .git/                    ← Git's database (don't touch this!)
    ├── objects/             ← Every version of every file
    ├── refs/                ← Pointers to branches
    ├── HEAD                 ← Your current branch
    ├── config               ← Your local settings
    └── ...other Git files...

The .git folder is where Git stores the entire history. You should never edit it manually — Git does everything automatically.

What Should Be In Git vs. What Shouldn't

✓ Include in Git ✗ Don't Include
Source code (.py, .js, .java) Passwords, API keys, .env files
README.md, documentation node_modules/, venv/, .vscode/
Project config files Compiled files (__pycache__, .o)
Database schemas Large binary files (videos, data)
Tests and test data IDE-specific settings

Special Files into Git

1.gitignore

You tell Git what to ignore using a special file called

The Simple Idea: Imagine you are packing a suitcase for a trip. You want your clothes, but you don't want to pack your household trash or your private diary. .gitignore is a checklist you give to Git that says, "Do not touch or pack these items." 

# .gitignore file
passwords.txt
config.env
node_modules/
__pycache__/
*.pyc
.vscode/
.idea/

2. .gitattributes

The .gitattributes file allows you to define attributes for paths in your repository. This is useful for specifying how Git should handle certain files, such as setting line endings or marking them as binary.

The Simple Idea: Imagine two people are writing a book together, but one uses a pencil and the other uses a pen. It creates a messy look. .gitattributes is a small rule book that forces everyone’s computer to format the code exactly the same way so it looks neat. 

# .gitattributes file
my-website/
├── .gitattributes      -- Put it in the root folder
├── .gitignore
├── index.html
└── logo.png
        
# 1. Handle line endings automatically for all text files
# This prevents Windows vs Mac formatting bugs
* text=auto

# 2. Explicitly force code files to use standard Linux/Mac line endings (LF)
*.js text eol=lf
*.html text eol=lf
*.css text eol=lf
*.json text eol=lf

# 3. Tell Git that images and archives are binary data
# This stops Git from accidentally trying to modify or merge them like text
*.png binary
*.jpg binary
*.zip binary

3. .gitkeep

Git does not track empty folders. If you want to keep an empty folder in your repository, you can add a placeholder file called .gitkeep inside it. This tells Git to include the folder even though it's empty.

The Simple Idea: Git is a mail carrier that only delivers packages with items inside. If you give Git an empty cardboard box, it throws it away because it thinks it's useless. But what if you need that empty box waiting at the destination? You put a tiny rock inside it so Git carries it. That rock is .gitkeep. 

my-website/
├── public/
│   └── uploads/
│       └── .gitkeep     -- This blank file forces Git to track the folder
├── index.html
└── styles.css
        
# 1. Navigate to your empty folder
cd public/uploads

# 2. Create the blank file
touch .gitkeep

Without Git: A bug appears in production. You have no idea when it started or who introduced it. Debugging nightmare.

HOW: How Git Actually Works Internally

The Three Stages (Most Important Concept)

Every file in Git exists in one of three states. Understanding these is the key to everything.

The Three Stages of Git:
Working Directory          Staging Area          Repository (.git)
  (Your laptop)         (Ready to save)       (Permanent history)
       |                      |                      |
  You edit files      You choose which       Git saves it
  freely here         files to save         forever here
       |                      |                      |
   git add           git commit
   Move ────────────→ ─────────────→

Stage 1: Working Directory

This is simply your project folder on your laptop. When you create a new file, edit existing code, or delete something — that happens here. Git can see the changes but hasn't saved anything yet.

Example: Rahul's Day

Rahul opens VS Code and creates login.py. He writes 50 lines of login code. This file is in the Working Directory. Git sees it but hasn't saved it.

Command to check: git status

Stage 2: Staging Area

The Staging Area (also called "Index") is a preparation zone. You choose which changes to include in the next save. This gives you fine control — maybe you changed 3 files but only want to save 2 of them right now.

Example: Ready to Save

Rahul's login.py is complete. He stages it:

git add login.py

Now it's in the Staging Area — ready to be saved, but not saved yet.

Think of it like preparing an envelope: You've decided what goes inside (staged), but you haven't sealed and mailed it yet (committed).

Stage 3: Repository (The Permanent History)

When you run git commit, Git permanently records a snapshot of all staged files. This is stored in the .git folder. From this point on, that version exists forever.

Example: Make It Permanent

Rahul commits:

git commit -m "Add login page with validation"

This snapshot is now saved forever in the repository. If Rahul breaks the code later, he can always come back to this version.

Complete Example: One Full Cycle

Terminal
# Rahul creates login.py (Working Directory)
$ cat > login.py << 'EOF'
def login(username, password):
return check_credentials(username, password)
EOF

# Check status
$ git status
On branch main
Untracked files:
login.py

# Stage the file (Staging Area)
$ git add login.py

# Check status again
$ git status
Changes to be committed:
new file: login.py

# Commit to repository (permanent save)
$ git commit -m "Add login functionality"
[main a1b2c3d] Add login functionality
1 file changed, 3 insertions(+)

# Check status one more time
$ git status
nothing to commit, working tree clean

How Commits Are Stored

A commit is NOT just the changes — it's a complete snapshot of your entire project at that moment. This is brilliant because:

  • You can instantly jump to any commit and see the project exactly as it was
  • No need to reconstruct from a chain of changes
  • Every version is independently accessible

Real Example: Complete Snapshots

Commit 1 (Day 1): Project has login.py (100 lines), database.py (50 lines)

Commit 2 (Day 3): Rahul edits login.py (120 lines). Priya adds schema.py (30 lines). Commit 2 stores: login.py (120 lines) + database.py (50 lines) + schema.py (30 lines) = complete snapshot

If you jump to Commit 1: Project has only login.py and database.py

If you jump to Commit 2: You get all three files with their latest versions

Branches: Parallel Timelines

A branch is simply a pointer to a commit. Think of it as a parallel universe where you can experiment without affecting the main timeline.

Branching in Action:
main branch (stable):
  Commit A ─── Commit B ─── Commit C
  (v1.0)       (v1.1)       (v1.2)

Rahul creates feature-login branch:
                            \
feature-login:             Commit D ─── Commit E
                           (add login)    (add validation)

Priya creates feature-payment branch:
                            \
feature-payment:           Commit F ─── Commit G
                           (add payment)  (add checkout)

Each person works independently. Main stays stable!

Merging: Bringing Work Together

When a feature is complete and tested, you merge it back into main. Git intelligently combines the changes.

Merging Complete:
Before Merge:
main:          Commit B ─── Commit C
                         \
feature-login:           D ─── E

After: git merge feature-login
main:          Commit B ─── Commit C ─── Commit M (merge)
                         \                /
feature-login:           D ─────────────E

Both branches now point to the merged result!

Handling Merge Conflicts

Sometimes two people edit the same file. Git asks you to decide which version to keep.

Real Conflict Example

Scenario: Priya and Rahul both edited config.py line 5:

  • Priya changed it to: DATABASE_HOST = "production-server"
  • Rahul changed it to: DATABASE_HOST = "localhost"

When Anaya tries to merge, Git shows both versions:

<<<<<<< HEAD
DATABASE_HOST = "production-server"
=======
DATABASE_HOST = "localhost"
>>>>>>> feature-dashboard

Anaya opens the file, decides which is correct (probably "localhost" for testing), removes the conflict markers, and commits: 

git add config.py
git commit -m "Resolve database host conflict"

Golden Rules for Team Work

✓ Rule 1: Never Push Directly to Main

Always create a feature branch. Push to your branch first. Let the team review before merging to main.

✓ Rule 2: Pull Before You Start

Always run git pull origin main before you start coding. This keeps you in sync with teammates.

✓ Rule 3: Commit Frequently

Commit every 30-60 minutes. Finished a function? Commit. Fixed a bug? Commit. More commits = finer history = easier debugging.

✓ Rule 4: Write Clear Commit Messages

Bad: "Fixed stuff"
Good: "Fix null pointer exception in attendance calculation when hours are zero"

Undoing Mistakes: Three Levels

Mistake Command Effect
Haven't staged yet git restore filename Discard changes in file
Staged but not committed git restore --staged filename Unstage the file
Already committed (local only) git reset HEAD~1 Undo the commit, keep changes
Already pushed to GitHub git revert commit-id Create a new commit that undoes it

SSH vs HTTPS: Which Should You Use?

HTTPS

Pros: No setup, works everywhere

Cons: GitHub asks for password every time

Best for: One-time use or unfamiliar computers

SSH

Pros: Set up once, never type password again

Cons: Takes 5 minutes to set up

Best for: Your own laptop (daily work)

Recommendation: Use SSH for your laptop. You'll feel like a professional developer when Git never asks for passwords again!

Alternatives: Other Version Control Systems

Git is the industry standard, but alternatives exist. Here's honest comparison:

Mercurial (Hg)

How it differs: Similar to Git but simpler syntax. Less distributed.

Used by: Some companies, but much less common.

Verdict: Not worth learning. Git dominates.

Subversion (SVN)

How it differs: Much older. Centralized model — history only on server.

Used by: Legacy systems, older enterprises.

Verdict: Dying out. You might encounter it, but Git is the future.

GitHub Alternatives: GitLab, Gitea, Bitbucket

GitLab

More features than GitHub, generous free tier, self-hostable

Good for: Teams wanting more control

Gitea

Lightweight, self-hosted only, minimal resource

Good for: Private company servers

Bitbucket

Integrates with Jira, free for small teams
